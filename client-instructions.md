## Creative Specs

### HTML5 Files

* **Supported file types:** HTML, HTM, JS, CSS, JPG, JPEG, GIF, PNG, JSON, XML, SVG, EOT, OTF, TTF, WOFF, WOFF2
* **Maximum number of files per .zip:** 100
* **Maximum size:** 10 MB
* HTML5 file should be zipped and include one HTML file and any other files that are references to the HTML
* **Animation:** :30s (looping included)

### Static Display

* **Most common sizes:** 160x600, 300x50, 300x250, 300x600, 320x50, 728x90
* **Image Types:** JPG, JPEG, GIF, PNG, SVG, WEB
* **Maximum file size:** under 150kb.

### Social (Meta)

#### Image

* **File Type:** JPG or PNG
* **Ratio:** 1.91:1 to 4:5
* **Resolution:**

  * **Headline:** 27 characters

* **Minimum Width:** 600 pixels
* **Minimum Height:**
  * **1:1 ratio:** 600 pixels

## Creative Asset Trafficking Timeline:

1. **Creative team notifies S&B** that raw assets, URLs, PDFs, etc. have been approved and shared.
   * S&B client services team will then have 2 days to process internally
2. **S&B Media Tech team has 3-5 days** to QA the new creative assets, upload into the needed platforms, and create vendor tags
   * *If we find any issues with the creatives that were handed off during our QA, this will affect timelines*
3. Once tags are created, **S&B client services team will have 2-3 days** to QA and send any necessary tags out to vendors
4. Vendors will typically have a **3-5 day turnaround time** from when tags are provided to then be able to launch

### Creative Trafficking QA

#### QA Steps Taken After Asset Handoff:

* These steps ensure assets are functioning as planned prior to launch
* Spring & Bond is responsible for the ad server trafficking, preview and delivering functioning tags to partners
* Creative agency is responsible for the build, creative functionality, and handoff of applicable trafficking details (ex. approved destination URL)
* If there are challenges with a creative asset, Spring & Bond and M+A work together to communicate barriers and work towards a resolution

#### Creative Asset QA:

1. **Check Asset Package:**
    a. All required sizes are present (for video duration is within specified limit)
    b. All assets are packaged to upload into ad server
    c. AFD PDFs
2. **Creative Files:**
    a. Veeva code within the creative matches creative asset tracker
    b. Imagery is clear
    c. Expiration date
3. **Functionality:**
    a. Destination URLs match creative tracker
    b. Destination URLs are appending with correct UTMs
    c. File size
    d. Frames are rotating for banners
4. **Post Launch:**
    a. Monitoring delivery to confirm media is live
    b. Screenshot deck within 72-hours

## Google Campaign Manager Banner Ad Technical Notes

**Last Updated:** July 08, 2024

**Ad Serving Platform:** Spring & Bond's primary ad platform is Google Campaign Manager (GCM). The following technical notes are meant to ensure that banners are coded to conform with GCM's creative requirements.

### Google HTML5 Validator

Banners must pass the HTML5 validator found here:

* [https://h5validator.appspot.com/dcm/asset](https://h5validator.appspot.com/dcm/asset)

### Google Documentation and Best Practices

Google maintains extensive documentation on creating HTML5 banners.

A good starting point for understanding some of the quirks of Campaign Manager is:

* ["How to prepare HTML5 display assets for Campaign Manager 360"](https://support.google.com/campaignmanager/answer/3145300)

Another good reference is:

* ["Get started with Studio: Build an HTML5 creative"](https://support.google.com/richmedia/answer/2672542)

### Managing Click Tags & UTM Params with GCM Creatives

GCM and the HTML5 validator have trouble finding click tags when they're set through multiple steps or asynchronous function calls.

Please do the following to ensure that click tags and UTM params work properly in your creatives:

1. **Use the Studio HTML5 SDK Enabler library**
   
   * ["HTML5 banner build guide: Add the HTML5 Enabler"](https://support.google.com/richmedia/answer/2672553)

2. **Statically declare a default clickTag variable in the main HTML file:**
   
   ```javascript
   <script type="text/javascript"> 
   var clickTag = "https://example.com"; 
   var clickTag2 = "https://example.com/pdfs/prescribing_information.pdf"; 
   </script>
   ```
   
   If there is more than one clickable URL, subsequent variables must be named `clickTag2`, `clickTag3`, etc. This naming convention is intentional and required for multiple exit URLs.

3. **Wrap the clickTag handlers in Enabler.exit() calls:**
   
   ```javascript
   function exitHandler(e) { 
   Enabler.exit( "clickTag", clickTag ); 
   }
   function exitHandler2() { 
   Enabler.exit( "clickTag2", clickTag2 ); 
   }
   ```

4. **Manually add the onClick handlers to the clickable elements**
   
   ```html
   <div id="bg_exit" 
   onClick="javascript:exitHandler();"></div> 
   
   ```

<div id="pi" 
  onClick="javascript:exitHandler2();"></div> 
```
Programmatically adding `onClick` event listeners does not always work. This approach has proven to be unreliable:

// DO NOT DO THIS 
$("#bg_exit").click( function() { exitHandler(); } );

Additional documentation on the Enabler library is here:
[https://www.google.com/doubleclick/studio/docs/sdk/html5/en/class_studio_Enabler.html](https://www.google.com/doubleclick/studio/docs/sdk/html5/en/class_studio_Enabler.html)
